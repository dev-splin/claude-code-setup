# 체계적 개발 프로세스

엄격한 18단계 개발 프로세스를 따릅니다. 체계적인 계획, 구현, 다층 검토를 통해 고품질 프로덕션 수준의 코드를 보장합니다.

## 모델 확인

스킬 시작 시 현재 모델을 확인하고, Opus가 아닌 경우 안내합니다:

```
⚠️ 계획 단계(1~4)는 Opus 4.6 모델 사용을 권장합니다.
   현재 모델이 Opus가 아니라면 `/model opus` 로 전환 후 다시 실행해주세요.
```

Opus로 확인되면 그대로 진행합니다.

## 실행 모드

시작 전 사용자에게 확인합니다:

> **실행 모드를 선택해주세요:**
> 1. **단계별 확인 모드** (기본) - 각 단계 완료 후 확인을 받고 다음 단계로 진행
> 2. **연속 실행 모드** - 주요 체크포인트에서만 확인하고 나머지는 연속 실행

사용자가 지정하지 않으면 **단계별 확인 모드**를 기본으로 사용합니다.

### 연속 실행 모드 체크포인트

연속 실행 모드에서는 다음 3개 단계 완료 후 반드시 사용자 확인을 받습니다:

1. **4단계 완료 후** (계획 단계 종료) - 최종 계획을 제시하고 구현 승인을 요청
2. **17단계 완료 후** (검토 단계 종료) - 배포 품질 최종 판정 결과를 보고하고 커밋 진행 여부 확인
3. **18단계 시작 전** (커밋/PR 작성) - 커밋 및 PR 작성 전 최종 확인

나머지 단계(1~3, 5~16)는 중단 없이 연속으로 실행합니다.

## 모델 사용 가이드

단계별로 적합한 모델을 사용합니다:

| 구간 | 단계 | 모델 | 이유 |
|------|------|------|------|
| 계획 | 1~4단계 | **Opus 4.6** | 깊은 분석과 계획 수립에 고성능 모델 활용 |
| 구현·검토 | 5~18단계 | **Sonnet 4.6** | 구현과 반복 검토에 빠른 모델 활용 |

4단계 체크포인트에서 구현 승인을 받을 때 사용자에게 모델 전환을 안내합니다:

```
💡 5단계부터는 Sonnet 4.6 모델로 전환하는 것을 권장합니다.
   `/model sonnet` 명령으로 전환해주세요.
```

## 진행률 표시

각 단계는 반드시 다음 형식의 진행률 헤더를 표시해야 합니다:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  [3/18] 검토 결과 재검토
  ████████░░░░░░░░░░░░░░░░░░░░░░ 17%
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

시작 시 TaskCreate로 18단계를 모두 생성하고, 각 단계 시작 시 TaskUpdate로 `in_progress`, 완료 시 `completed`로 표시합니다.

## 세션 복구

18단계를 시작하기 전에, 프로젝트 루트에 `task_plan.md`가 존재하는지 확인합니다.
파일이 있으면 `task_plan.md`, `findings.md`, `progress.md`를 읽고 중단된 단계부터 이어서 진행합니다.

## 18단계

### 1단계: 계획 수립
**목표**: dev-process-planner 서브에이전트를 사용하여 계획 파일을 생성합니다.

dev-process-planner 서브에이전트에 다음을 전달하여 계획 수립을 위임합니다:
- `user_request`: 사용자의 원본 요청
- `project_root`: 현재 프로젝트 루트 경로

서브에이전트가 완료되면 생성된 `task_plan.md`를 읽어 계획을 확인합니다.

**단계별 모드**: task_plan.md의 내용을 사용자에게 제시하고 승인을 요청합니다.

---

### 2단계: 계획 검토
**목표**: 수립한 계획의 타당성을 검토합니다.

`task_plan.md`를 읽어 계획을 확인한 뒤 검토합니다:
- 계획이 요구사항을 모두 충족하는지 확인
- 빠뜨린 부분이 없는지 검토
- 기술적 실현 가능성 평가
- 더 나은 대안이 있는지 검토

검토 결과 수정이 필요하면 `task_plan.md`를 직접 업데이트합니다.

**단계별 모드**: 검토 결과를 사용자에게 보고합니다.

---

### 3단계: 검토 결과 재검토
**목표**: 2단계의 검토가 정확한지 다시 한 번 확인합니다.

`task_plan.md`를 다시 읽어 2단계 검토 결과를 재확인합니다:
- 검토 과정에서 놓친 부분이 없는지 확인
- 검토 의견의 논리적 타당성 재확인
- 필요시 task_plan.md 수정

**단계별 모드**: 최종 검토 결과를 사용자에게 보고합니다.

---

### 4단계: 과도한 계획 여부 검토
**목표**: 계획이 과도하거나 불필요하게 복잡하지 않은지 검토합니다.

`task_plan.md`를 읽어 계획을 확인한 뒤 검토합니다:
- YAGNI 원칙 적용 - 당장 필요하지 않은 것은 제거
- 오버엔지니어링 요소 식별
- 최소한의 변경으로 목표를 달성할 수 있는지 확인
- 불필요한 추상화나 일반화가 포함되어 있지 않은지 확인

조정 사항이 있으면 `task_plan.md`를 업데이트합니다.

**단계별 모드**: 조정된 최종 계획을 사용자에게 제시하고 구현 승인을 요청합니다.
**연속 실행 모드**: 🔒 **체크포인트** - 최종 계획을 제시하고 구현 승인을 요청합니다. 승인 없이 5단계로 넘어가지 않습니다.

---

### 5단계: 구현
**목표**: 승인된 계획에 따라 코드를 구현합니다.
- 계획의 각 항목을 순서대로 구현하며, 항목 완료마다 간략히 진행 상황 표시
- 기존 코드 패턴과 컨벤션 준수
- 각 변경 사항의 목적을 명확히 유지
- 관련 테스트가 있으면 함께 수정/추가
- 프로젝트에 린트/타입체크 설정이 있으면 구현 완료 후 실행하여 오류 확인

**단계별 모드**: 구현 완료 후 변경 사항 요약을 보고합니다.

---

### 6단계: 구현 목적 부합 검토
**목표**: 구현이 원래 목적에 맞게 잘 되었는지 검토합니다.
- 원래 요구사항과 구현 결과 대조
- 누락된 기능이 없는지 확인
- 의도하지 않은 동작 변경이 없는지 확인

**단계별 모드**: 검토 결과를 보고합니다.

---

### 7단계: 잠재적 이슈 검토
**목표**: 버그, 보안 문제, 크리티컬 이슈를 점검합니다.
- 잠재적 버그 식별 (edge cases, null checks, race conditions 등)
- 보안 취약점 검토 (injection, XSS, 인증/인가 등)
- 성능 이슈 검토
- 에러 핸들링 적절성 확인

**단계별 모드**: 발견된 이슈와 수정 제안을 보고합니다. 이슈가 있으면 수정 후 진행합니다.

---

### 8단계: 개선 내용 검증
**목표**: 7단계에서 수정한 내용이 새로운 문제를 만들지 않았는지 확인합니다.
- 수정 사항이 기존 기능을 깨뜨리지 않는지 확인
- 수정으로 인한 새로운 edge case 점검
- 수정 전후 동작 비교
- **새로운 문제가 발견되면 7단계로 돌아가 수정 후 다시 8단계를 수행합니다 (최대 3회 반복)**

**단계별 모드**: 검증 결과를 보고합니다.

---

### 9단계: 대형 함수/파일 분리
**목표**: 과도하게 큰 함수나 파일이 있다면 적절한 크기로 분리합니다.
- 함수 길이와 복잡도 점검
- 파일 크기와 책임 범위 점검
- 단일 책임 원칙에 따라 분리 필요 여부 판단
- 필요시 리팩토링 수행

**단계별 모드**: 분리 결과 또는 분리 불필요 판단을 보고합니다.

---

### 10단계: 코드 통합 및 재사용 검토
**목표**: 기존 코드와 통합하거나 재사용할 수 있는 부분이 있는지 검토합니다.
- 중복 코드 식별
- 기존 유틸리티/헬퍼 함수 활용 가능성 확인
- 공통 패턴 추출 가능성 검토
- 필요시 리팩토링 수행

**단계별 모드**: 검토 결과를 보고합니다.

---

### 11단계: 사이드 이펙트 확인
**목표**: 변경 사항으로 인한 사이드 이펙트가 없는지 확인합니다.
- 변경된 함수/모듈을 사용하는 다른 코드 확인
- 의존성 체인 추적
- 전역 상태 변경 영향 확인
- API 계약 변경 여부 확인

**단계별 모드**: 사이드 이펙트 분석 결과를 보고합니다. 이슈가 있으면 수정 후 진행합니다.

---

### 12단계: 전체 변경 사항 재검토
**목표**: 지금까지의 모든 변경 사항을 처음부터 다시 한 번 검토합니다.
- 모든 변경 파일을 순서대로 검토
- 변경 사항 간의 일관성 확인
- 전체적인 흐름이 자연스러운지 확인

**단계별 모드**: 전체 검토 결과를 보고합니다.

---

### 13단계: 불필요한 코드 정리
**목표**: 구현 과정에서 불필요해진 코드를 정리합니다.
- 사용하지 않는 import 제거
- 사용하지 않는 변수/함수 제거
- 불필요한 주석 제거
- dead code 정리
- 임시 코드/디버깅 코드 제거

**단계별 모드**: 정리 결과를 보고합니다.

---

### 14단계: 코드 품질 검토
**목표**: 코드 품질이 충분히 높은지 검토합니다.
- 네이밍 컨벤션 일관성
- 코드 가독성
- 적절한 추상화 수준
- 타입 안전성
- **프로젝트에 설정된 테스트 실행** (npm test, pytest 등) — 실패 시 즉시 수정
- **린트/포맷팅 검사 실행** (eslint, prettier, ruff 등) — 위반 시 즉시 수정
- **빌드/타입체크 실행** (tsc, mypy 등 해당시) — 실패 시 즉시 수정

**단계별 모드**: 품질 검토 결과를 보고합니다.

---

### 15단계: 사용자 흐름 검증
**목표**: 실제 사용자의 사용 흐름에서 문제가 없는지 확인합니다.
- 주요 사용 시나리오 시뮬레이션
- 에러 상황에서의 사용자 경험 확인
- UI/UX 관점에서의 자연스러운 흐름 확인 (프론트엔드)
- 접근성 및 사용성 점검 (프론트엔드)
- API 호출 시나리오 검증 (백엔드)
- 라이브러리의 경우 공개 API 사용 예시로 검증
- **사용자 흐름이 해당되지 않는 변경(내부 리팩토링, 설정 변경 등)은 해당 없음으로 표시하고 넘어갑니다**

**단계별 모드**: 사용자 흐름 검증 결과를 보고합니다.

---

### 16단계: 연관 부분 심층 검토
**목표**: 검토 과정에서 발견된 잠재적 문제와 관련된 모든 부분을 철저히 검토합니다.
- 이전 단계들에서 발견된 모든 우려 사항 재점검
- 관련 코드 경로 추적 및 검증
- 발견된 패턴이 다른 곳에서도 문제를 일으키지 않는지 확인
- 필요시 추가 수정 수행

**단계별 모드**: 심층 검토 결과를 보고합니다. 수정 사항이 있으면 다시 검증합니다.

---

### 17단계: 배포 품질 최종 검토
**목표**: 이대로 배포해도 될 수준의 품질인지 최종 확인합니다.
- 모든 변경 사항 최종 점검
- 프로덕션 환경에서의 동작 예상
- 롤백 가능성 확인
- 모니터링/로깅 적절성 확인
- **최종 판정: 배포 가능 / 추가 수정 필요**

**단계별 모드**: 최종 판정 결과를 보고합니다. 추가 수정이 필요하면 수정 후 재검토합니다.
**연속 실행 모드**: 🔒 **체크포인트** - 최종 판정 결과를 보고하고 커밋 진행 여부를 확인합니다.

---

### 18단계: 커밋 및 PR 작성
**목표**: 변경 사항을 커밋하고, 필요시 PR을 작성합니다.
- 변경 사항을 적절한 단위로 커밋
- 명확한 커밋 메시지 작성
- PR이 필요한 경우: PR 제목과 설명 작성 (변경 사항 요약, 테스트 계획 포함)
- **사용자에게 커밋만 할지, PR도 작성할지 확인합니다**

**단계별 모드**: 커밋과 PR 작성 전 사용자에게 최종 확인을 받습니다.
**연속 실행 모드**: 🔒 **체크포인트** - 커밋 메시지와 PR 내용을 제시하고 사용자 승인 후 실행합니다.

---

## 단계별 확인 (단계별 확인 모드)

단계별 확인 모드에서는 각 단계 완료 후 다음을 표시합니다:

```
✓ [N/18] {단계명} 완료

다음 단계: [{N+1}/18] {다음 단계명}
계속 진행할까요? (y: 진행 / n: 중단 / skip: 건너뛰기 / auto: 이후 연속 실행)
```

- **y** 또는 긍정 응답: 다음 단계로 진행
- **n** 또는 부정 응답: 중단하고 추가 지시를 기다림
- **skip**: 다음 단계를 건너뛰고 그 다음 단계로 이동
- **auto**: 나머지 단계를 연속 실행 모드로 전환

## 중요 규칙

1. **단계를 암묵적으로 건너뛰지 않기** - 모든 단계는 명시적으로 실행되거나 사용자가 명시적으로 건너뛰어야 합니다
2. **수정 전 반드시 읽기** - 기존 코드를 변경하기 전에 항상 먼저 읽습니다
3. **기존 패턴 유지** - 프로젝트에서 확립된 컨벤션을 따릅니다
4. **문제 발견 즉시 수정** - 검토 단계에서 문제가 발견되면 다음으로 넘어가기 전에 수정합니다
5. **철저하되 실용적으로** - 검토는 피상적인 체크리스트가 아닌 실질적인 내용이어야 합니다
6. **되돌아가기** - 검토 단계(6~16)에서 심각한 설계 결함이 발견되면 5단계(구현)로, 요구사항 자체의 문제라면 1단계(계획)로 돌아갑니다. 되돌아갈 때는 사유를 사용자에게 보고합니다.
7. **테스트 우선** - 프로젝트에 테스트/린트/빌드 설정이 있다면 반드시 실행하여 검증합니다
8. **progress.md 업데이트** - 각 단계 완료 시 progress.md에 완료 내용을 기록합니다
